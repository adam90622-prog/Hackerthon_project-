# 회고록 — Problem 04 다이버즈 SNS 콘텐츠 3종 동시 자동 작성

## 0. 프로젝트 한 줄 요약 + 문제 정의
간호사 커리어 플랫폼 '널스빌리지'에서 마케터 1인이 건당 2~3시간 걸리던 카드뉴스·카카오
메시지·뉴스레터 3종 제작을, 현장 용어·브랜드보이스를 반영해 재사용 가능한 파이프라인으로
자동화하고 5분 내 초안 생성이 가능하도록 풀었습니다. 문제는 속도(건당 2~3시간)·일관성
(담당자 교체 시 브랜드보이스 흔들림)·시의성 대응(이벤트 대응 지연) 세 가지 병목으로
정의했습니다.

## 1. 문제를 어떻게 정의했는가
Notion 원문 스펙(①기업소개~⑦self-review)을 기준점으로 삼고, 1차 산출물과 항목별로
대조하는 진단표를 먼저 만들었습니다. Basic 채점 기준(3종 포맷 40점·현장 용어 20점·
타깃톤일치 30점·제출형식 10점), Standard 가산(파이프라인 재현성 15점·캘린더 설계 15점),
Challenge 가산(전략안 실효성 20점·양식설계 10점)을 각각의 요구 문장 그대로 다시 읽고,
"이 항목이 요구하는 것"과 "실제로 만든 것"을 나란히 놓고 비교하는 방식으로 문제를
재정의했습니다.

## 2. 무엇을 만들었는지 (Basic → Standard → Challenge)

**Basic**: 18개 시드 전체에 카드뉴스(4장, 장별 헤드라인+본문+CTA)·카카오메시지(200자
이내)·뉴스레터(550~650자) 3종을 생성하고 현장 용어 태그를 정리했습니다. 실무 의미는
"마케터 1인이 혼자 2~3시간 걸리던 초안 작성을, 톤·타깃 일관성을 유지한 채 몇 분 안에
끝낼 수 있다"는 것을 실제 18건으로 증명한 것입니다. 결측(id12)·중복(id11/17) 같은
실무형 데이터 함정도 추론 근거를 남기며 처리했습니다.

**Standard**: `clean_seeds.py → check_cadence.py → run_pipeline.py → validate_content.py`
로 이어지는 재사용 파이프라인과, 발행 캘린더·채널별 핵심 지표·브랜드보이스 가이드를
만들었습니다. 실무 의미는 "새 시드가 오는 다음 주에도 같은 절차·같은 품질로 콘텐츠가
나온다"는 재현성과, "언제·어떤 지표로 운영할지"에 대한 실행 가능한 기준을 남긴 것입니다.

**Challenge**: `check_cadence.py`가 드러낸 세그먼트별 상시 콘텐츠 부족(전략안 1, P0)과,
ref_event 8/18건 분포에서 나온 시의성 대응 구조화(전략안 2, P1)를 문제정의/근거데이터/
전략제안/우선순위 4단 형식으로 제안했습니다. 실무 의미는 "지금 데이터가 보여주는 다음
액션이 무엇인지"를 마케터가 아니라 콘텐츠 전략가 관점에서 제시한 것입니다.

## 3. 후속 캠페인 기획안
Challenge 전략안 1을 후속 캠페인의 뼈대로 그대로 씁니다. `check_cadence.py` 실측 기준
신규RN·간호학생 k=3(70÷3=23.3일), 경력RN k=4(70÷4=17.5일)로 세 세그먼트 모두 목표
캐던스(14일)에 못 미칩니다. `70÷k≤14`를 만족하려면 k≥5가 필요하므로, 다음 캠페인에서는
신규RN·간호학생에 상시(evergreen) 소재 2개씩, 경력RN에 1개를 추가해 각 세그먼트 k=5를
채우는 것을 1차 목표로 잡습니다. 소재 방향은 세그먼트 특성에서 자연스럽게 도출됩니다
(신규RN — 동료 갈등 대처, 경력RN — 번아웃 자가진단 등). 이후 `check_cadence.py`를
재실행해 PASS 전환을 확인하는 것으로 캠페인 성공을 검증할 수 있습니다.

## 4. 재현 가능한 Skill 체인 구성
```
/analyze  → 시드 CSV 파악(결측·분포·용어 사전 숙지)
/insight  → 채널별 전략 + 현장 용어 활용 계획 수립
/generate → 3종 콘텐츠 세트 생성 (output_content_set.md)
/review   → 글자수·용어 반영 자가 점검
/strategize → 캐던스·ref_event 패턴 기반 발행 전략안 생성 (Challenge, 신규)
```
스크립트 체인: `clean_seeds.py`(정제) → `check_cadence.py`(캐던스 점검) →
`run_pipeline.py`(둘을 순서대로 실행하는 래퍼) → `validate_content.py`(생성물 형식 검증)
→ `measure_kpi.py`(KPI 4종 실측) → `build_report.py`(콘텐츠+KPI를 정적 HTML로 렌더링).
새 seeds CSV 하나만 있으면 이 체인 전체가 동일한 순서로 재현됩니다.

## 5. 주요 설계 결정 (decisions.md 기준)
- **id12 키워드 결측**: topic·target·tone 맥락 추론으로 "국시,면허,RN" 확정, 재실행해도
  같은 값이 나오도록 고정(MANUAL_OVERRIDES)
- **id11/17 중복 topic**: "당하는 1인칭(간호학생)" vs "돌아보는 선배 시점(경력RN)"으로
  관점을 완전히 분리 — 단순 톤 변경이 아니라 서술 시점 자체를 다르게 설계
- **방향을 바꾼 결정**: 캐던스 계산(70÷k)이라는 통찰 자체는 유효했지만, 이를 근거로
  발행 이력 시뮬레이션 → 신규 기사 자동분류 → 미래 재스케줄링 → 대시보드 UI까지 확장
  구축했던 것을 스펙 재대조 과정에서 전량 롤백했습니다. 통찰(k값 부족)은 버리지 않고
  Challenge 전략안의 근거 데이터로 이관해 "인사이트는 살리고 과잉 구현은 걷어내는" 방향으로
  틀었습니다.
- **KPI 지표명 조정**: 최초 "오생성율"로 설계했다가 "생성 성공률"로 변경 — 실패율보다
  성공률이 문제 해결도를 더 직관적으로 보여준다는 판단.

## 7. 힘들었던 점과 해결 방법
- **범위가 스스로 부풀었던 문제**: Standard 캘린더 설계 중 나온 통찰(캐던스 부족)에
  꼬리를 물고 발행 이력 시뮬레이션·기사 자동분류·미래 재스케줄링·대시보드까지 만들면서,
  정작 배점이 가장 큰 Challenge 전략안(20점)이 비어 있는 역설적 상황이 됐습니다. →
  Notion 원문 스펙을 기준점으로 놓고 항목별 진단표를 만들어 재정렬했습니다.
- **이미 제출된 상태에서 수정해야 하는 제약**: GitHub 링크·Streamlit 데모·발표자료 링크가
  이미 제출돼 있어 완전히 새로운 산출물을 만들 수 없었습니다. → 기존 구조(app.py가
  dashboard.html을 읽는 구조)는 그대로 두고, 그 안의 콘텐츠만 스펙에 맞게 교체하는
  방식으로 해결했습니다.
- **세션 간 컨텍스트 단절**: 토큰 제약으로 매 단계 새 Claude Code 세션을 열어야 해서
  이전 대화 맥락이 이어지지 않았습니다. → 각 단계마다 스스로 완결된("이 프롬프트만
  읽으면 실행 가능한") 확인용 프롬프트를 설계했고, 특히 리스크가 큰 스크립트(HTML 리포트
  생성)는 제가 먼저 실제 데이터로 테스트한 뒤 검증된 코드를 프롬프트에 그대로 넣어
  오류 재현 가능성을 낮췄습니다.
- **프롬프트 포맷 문제**: 프롬프트 안에 마크다운 코드블록(```)이 중첩되면서 바깥쪽
  블록이 먼저 닫혀 복사가 깨지는 문제가 있었습니다. → 바깥쪽을 4개 백틱으로 감싸는
  방식으로 해결했습니다.

## 8. 제약, 금지 범위
Notion 스펙이 명시한 제외 범위: 카드뉴스 이미지 자동 생성, 실제 SNS 자동 발행, 영상/
릴스/숏폼, 콘텐츠 성과 분석, 의료 정보 팩트체크, 실제 운영 데이터 사용.
1차 산출물 재점검 과정에서 추가로 걷어낸 범위: 대시보드/웹 UI 신규 구현, 신규 기사
자동분류 시스템, 발행 이력 시뮬레이션 및 미래 재스케줄링, 합성 더미 데이터(50행) 생성.
다만 이미 제출된 Streamlit 데모(app.py) 구조 자체는 유지했고, 그 안의 콘텐츠만 스펙
범위로 교체했습니다.
